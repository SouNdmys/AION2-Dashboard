# PROJECT_REVIEW

## 1. 项目概览

项目名称：`AION2 Dashboard`

项目定位：一个离线桌面工具，帮助 AION2 玩家管理多账号多角色周常/日常进度，并在做装时同步处理抓价、缺料、利润和现金换算。

项目类型：学习项目 + 兴趣项目 + 真实自用工具。

它不是一个为了展示技术而做的 demo，而是一个有明确用户场景、有版本迭代、有打包发布、有自动更新、有热修节奏的完整软件项目。

当前收口版本：`v2.2.4`

当前维护策略：低频维护。后续有明确新需求或规则热修再继续开发，不再无边界扩功能。

## 2. 为什么做这个项目

最初问题很直接：AION2 这种多角色、多周常、多资源消耗的游戏，只靠脑子记和表格记，维护成本太高。尤其是：

- 多个角色的每日/每周任务很容易漏
- 奥德、回廊、票券这类资源容易算错
- 做装备时材料、库存、市场价格、利润判断要来回切信息

这个题目适合做第一款完整程序，有几个原因：

- 用户场景真实，自己能长期使用
- 功能是离散但能组成闭环的，适合迭代
- 可以同时练到架构、UI、打包、更新、文档、测试
- 做完以后，结果不是“练手代码”，而是一个真能用的工具

## 3. 核心用户与使用场景

核心用户首先是自己，其次是和自己有类似玩法的小圈层玩家。

高频使用场景有两个：

1. 多角色任务管理
- 看今天/这周哪些任务没做
- 快速录入完成次数
- 判断哪个角色优先做什么

2. 做装模拟
- 看配方需要什么材料
- 看库存缺口
- 看当前价格下值不值得做
- 估算补齐材料还要收多少金币/RMB

低频但关键的场景：

- 游戏规则变了，需要快速热修
- 要发新版本，需要稳定打包和自动更新
- OCR 或价格系统出问题时，需要能快速排查

## 4. 产品边界

这个项目最终保留的主线是两条：

1. 多角色任务管理
2. 做装模拟与市场辅助

明确做的内容：

- 账号/角色管理
- 任务录入与优先级排序
- 周常/日常/回廊/奥德等规则驱动的任务管理
- OCR 抓价
- 价格历史与市场分析
- 做装利润模拟
- 库存、缺料、采购清单、现金换算
- 安装包发布与自动更新

明确不做或主动放弃的方向：

- 账号级复杂资源统筹
- 版本规则配置化系统
- DPS 插件
- 内存读数
- 逆向第三方插件

这些方向没做，不是因为“做不到”本身，而是因为不值得：

- 和主线不一致
- 会显著增加复杂度
- 维护成本过高
- 一旦滑向灰区，项目性质会被带偏

这个项目最正确的边界判断，是没有继续横向乱长。

## 5. 版本演进与关键里程碑

### 5.1 初始可用阶段

最早的阶段重点是把“能用的闭环”做出来，而不是把结构做漂亮。

当时最小闭环是：

- 多角色基础数据
- 日常/周常任务状态
- 能量与回廊规则
- 基本录入

当时的主要问题也很明显：

- UI 信息很多，但层级不清楚
- 状态逻辑逐渐堆积
- 代码开始变重
- 发布链路还不稳

### 5.2 架构重构阶段

后来必须做架构重构，不是为了“技术洁癖”，而是因为继续堆功能已经开始伤可维护性。

重构前的症状：

- renderer 层逻辑越来越集中
- workshop 相关逻辑耦合度高
- shared 契约不够集中
- 改一个地方容易带出别的问题

重构后的收益：

- `main / preload / renderer / shared` 职责更清楚
- workshop 相关逻辑按 domain 拆开
- IPC 契约更稳定
- 热修开始能在局部落地，而不是到处改

这次重构的价值不在“结构更漂亮”，而在于后面几个月的热修都能稳住。

### 5.3 热修与发布阶段

这个阶段暴露出很多真实的软件工程问题：

- OCR 安装版模型路径问题
- 自动更新 `latest.yml` 哈希不一致
- 手动价格与 OCR 价格逻辑冲突
- 游戏规则改变后，任务模型需要快速调整
- 安装版更新体验不够自然，用户感知更像“重新装一遍”

这些问题逼出了几个非常关键的工程动作：

- 每个热修尽量单独 commit
- 每个热修后跑 `typecheck + test:unit`
- 发布前固定跑 `test:smoke`
- 增加 `latest.yml` 校验脚本
- 把“后台下载、退出后安装”升级成“启动先更新、静默安装后再进主界面”

如果没有这些流程，后面发版会越来越不可信。

后期一个很关键的体验升级，是把安装版更新链路改成更接近常见桌面客户端的方式：

- 启动先检查更新
- 如果发现更新，主界面先不进入
- 启动页直接显示“正在更新到 vX.Y.Z...”
- 下载完成后静默安装
- 自动重启进入新版
- 如果更新失败，继续进入旧版，不锁死应用

这一步的价值不只是“好看一点”，而是把更新从“用户手动重新安装”的感受，改成了“应用自己完成升级”的感受。  
它标志着这个项目在发布体验上也真正进入了“完整软件”阶段。

### 5.4 UI 收口阶段

UI 收口是这个项目后期最重要的一次产品调整。

旧 UI 的问题不是“不好看”那么简单，而是：

- 主任务和高级工具混在同一层
- 页面打开后先看到控件，不是先看到任务
- 角色页和工坊页信息密度过高
- 高级能力没有后撤，用户被迫处理太多信息

最终新 UI 的核心原则变成了三条：

- 主任务优先
- 高级工具后撤
- 信息层级克制

这一步非常关键。因为它把项目从“功能集合”拉回了“用户工作台”。

## 6. 架构复盘

### 6.1 当前分层

当前结构是对的，至少足以支撑这个项目继续稳定热修。

`src/main`
- 持久化
- 领域规则
- workshop 数据、价格、OCR、simulation
- 打包后真实运行环境相关逻辑

`src/preload`
- 提供安全桥接 API
- 控制 renderer 能访问什么

`src/renderer`
- 页面
- 组件
- 交互
- UI 状态

`src/shared`
- 类型
- 常量
- 规则
- IPC 契约

这个分层的实际价值在于：功能增加后，依然能判断“这段逻辑该放哪里”。

### 6.2 哪些架构决策是对的

最值得保留的决策：

- 把共享类型与规则集中在 `shared`
- 明确主进程与渲染层边界
- workshop 的计算逻辑和界面逻辑分开
- 发布脚本、校验脚本独立出来
- 文档和操作日志跟着版本走

这些决定没有带来炫技效果，但直接降低了后期改动成本。

### 6.3 哪些地方以后可以更好

仍然存在的不足：

- renderer 里仍有部分组件偏大
- 有些页面的 view model 还是偏重
- 某些命名是迭代过程中逐步形成的，不是一次性设计好的
- 浅色主题虽然已经收敛，但还不是完全 token-only

这些问题都不是致命问题，但下一个项目可以更早处理。

## 7. 最有价值的产品决策

几个关键决策，决定了这个项目后期没有失控：

### 7.1 做装模拟成为第二主线

这一步很值。因为它不是无关功能，而是和玩家真实行为紧密相连：

- 先管理任务
- 再管理做装与市场判断

这两条线都属于“角色运营工具”，没有偏题。

### 7.2 市场工具后撤

把 OCR、市场分析、历史价格、库存放进折叠区，是非常重要的 UI 决策。

因为用户主需求不是“研究工具”，而是“我这件装备现在做不做”。

### 7.3 材料流放到结果区顶部

最终把结果区顺序调整成：

1. 材料明细与库存修正
2. 材料缺口与采购清单
3. 模拟结论
4. 金价换算辅助

这是符合真实操作顺序的：

- 先补材料和价格
- 再看缺口
- 再看利润
- 最后估算现金成本

### 7.4 不再继续开新系统

后期没有继续做账号级资源统筹、版本配置化、DPS 插件之类的能力，这是正确的。

因为项目真正成熟，不是“功能越来越多”，而是“边界越来越清楚”。

## 8. 最关键的问题与踩坑

### 8.1 技术坑

1. OCR / ONNX 安装版路径问题  
打包后 `app.asar` 和 `app.asar.unpacked` 的路径差异，导致 OCR 在安装版直接失效。

2. 自动更新哈希问题  
`latest.yml` 和实际安装包哈希不一致，会导致更新检查失败。

3. 自动更新体验设计问题  
如果只是“检查更新 -> 下载 -> 退出再装”，用户心理感受仍然像重新安装。  
真正自然的体验，需要把更新检查、安装时机、失败回退和启动屏一起考虑。

4. 手动价格和 OCR 价格冲突  
如果状态归并不严谨，手动输入很容易被自动值覆盖。

5. UI 主题残留  
做浅色主题时，旧的深色 utility 残留会让界面出现局部不协调。

### 8.2 流程坑

1. 未冻结结构就持续堆需求，代码会开始变形  
2. 没有自动校验时，发版元数据很容易出错  
3. UI 如果不分阶段推进，很快会从“改进”变成“混乱”  
4. 如果没有操作日志，几周后会很难回忆“为什么当时要这么改”

## 9. 这次项目做对了什么

最值得肯定的不是某个功能，而是开发方法：

- 每个热修尽量单独 commit
- 每次功能落地后及时回归
- 发版流程被脚本化
- 重要版本有 release notes
- UI 改造分 phase 推进
- 文档不是最后补，而是过程里持续更新
- 安装版更新体验被纳入产品设计，而不是只停留在“能更新就行”

这些方法决定了项目能一直做下去，而不是做几周就炸。

## 10. 如果重来一次

如果从头再做一次，我会更早做这几件事：

- 更早冻结命名和页面边界
- 更早定义“哪些绝对不做”
- 更早补自动更新校验
- 更早让 UI 只围绕主任务展开
- 更早建立统一 token 系统

但即便没有一开始就做到，这个项目后期还是把这些关键点补回来了，这是成熟度的一部分。

## 11. 这个项目真正带来的能力增长

这项目最值钱的产出，不只是软件本身，而是能力：

- 需求收敛
- 架构分层
- 状态逻辑控制
- 热修流程
- 自动更新与发布
- 安装版更新体验设计
- UI 信息架构设计
- 与 AI 协作开发

这些能力未来做别的项目还能继续用。

## 12. 可迁移经验

以后做任何新项目，都值得继续沿用这些经验：

- 先做最小闭环
- 架构冻结后再做热修
- 发布必须脚本化
- 更新策略要尽早设计，不要等到最后才补
- 大 UI 改造必须分阶段
- 文档要和代码一起演进
- 主任务必须在页面最前面
- 低频工具必须后撤

## 13. 后续维护策略

这个项目后面适合的维护方式是：

- 游戏规则变更时做热修
- 确定高价值且不破坏边界的新功能再做
- 不再为“看起来还能做很多”而继续扩张

如果未来不玩这个游戏了，这个项目完全可以停止维护。  
这不是失败，而是兴趣项目的正常生命周期。

## 14. 最终结论

从“人生第一款程序”的标准来看，这个项目已经足够称得上完整软件项目。

它具备：

- 真实问题
- 明确用户
- 多轮迭代
- 架构重构
- UI 收口
- 测试
- 打包
- 自动更新
- 发布与热修

这意味着你已经不是“只会拼代码”，而是已经完整经历了一次软件产品从 0 到稳定版的过程。

这个项目以后即使不再继续长，也已经值回票价。真正留下来的，是你以后做任何项目都会继续用到的判断力和方法论。
