# MY_DEV_PLAYBOOK_OUTLINE

## 1. 这份 Playbook 的用途

- 目标：
  - 以后做新项目时，直接按这份流程推进
- 适用范围：
  - 学习项目
  - 自用工具
  - 小型桌面应用

## 2. 新项目启动前

### 2.1 先回答这 5 个问题

- 这个项目到底解决什么问题：
- 谁会真正使用它：
- 最小可用版本必须包含什么：
- 明确不做什么：
- 什么状态算“项目完成”：

### 2.2 先写的 4 份文件

- `README.md`
- `FRAMEWORK.md`
- `OPERATION_LOG.md`
- `RELEASE_RUNBOOK.md`

## 3. 我以后默认采用的开发原则

- 先闭环，再扩功能
- 先收边界，再谈架构优雅
- 先保可发布，再保功能丰富
- 不在主流程里塞低频工具
- 文档和日志要跟代码一起演进

## 4. 项目结构基线

### 4.1 推荐目录分层

- `src/main`
- `src/preload`
- `src/renderer`
- `src/shared`
- `scripts`

### 4.2 各层职责约束

- 主进程处理：
  - 持久化
  - IPC 注册
  - 系统资源访问
- 渲染层处理：
  - 界面
  - 交互
  - 轻量 view model
- 共享层处理：
  - 类型
  - 常量
  - 规则
  - 契约

## 5. 需求推进方法

### 5.1 功能开发顺序

1. 明确用户动作链路
2. 找主任务
3. 把低频能力后撤
4. 确定最小可用实现
5. 再开始编码

### 5.2 什么时候该冻结结构

- 架构重构刚做完
- 功能连续迭代中
- 发布窗口临近

冻结后规则：
- 不再做结构性重构
- 只做功能热修或局部增强
- 每次改动尽量小 commit

## 6. 热修流程

- 每个热修只解决一个明确问题
- 改完必须跑：
  - `npm run typecheck`
  - `npm run test:unit`
- 发版前必须再跑：
  - `npm run test:smoke`
- 发布说明必须写清：
  - 改了什么
  - 为什么改
  - 影响范围

## 7. UI 工作方法

### 7.1 UI 不先谈“好看”

- 先看信息架构对不对
- 先看主任务是否突出
- 先看高级工具是否后撤

### 7.2 UI 改造顺序

1. 设计 token 收敛
2. 信息架构重排
3. 结果区优先级整理
4. 侧栏与导航归位
5. 全局一致性收尾
6. 最后才做视觉抛光

### 7.3 判断 UI 是否变好的标准

- 打开页面时先看到主任务
- 不需要读很多字就能开始操作
- 高优先信息能被一眼扫到
- 低频功能不抢视线

## 8. 发布流程

### 8.1 标准检查

- `npm run typecheck`
- `npm run test:unit`
- `npm run build`
- `npm run test:smoke`
- `npm run dist:win`
- `npm run check:release-yml`

### 8.2 发布资产检查

- Setup
- Portable
- 7z
- blockmap
- latest.yml

### 8.3 发布后验证

- 安装版能启动
- 版本号正确
- 自动更新能识别
- 哈希一致

## 9. 文档纪律

- 大功能要有说明文档
- 每次发版要有 release notes
- 每个阶段要补 operation log
- 重要规则必须有基线文件

## 10. 和 AI 协作的方法

### 10.1 适合交给 AI 的事情

- 代码实现
- 结构梳理
- 回归执行
- 文档整理
- 发布流程执行

### 10.2 不该模糊表达的事情

- 版本边界
- 用户优先级
- 哪些功能绝对不做
- 发布策略

### 10.3 高质量协作方式

- 先给明确目标
- 再给边界
- 再给验收标准
- 让 AI 按小步提交推进

## 11. 我以后要避免的坏习惯

- 一边重构一边堆功能
- 没测就发版
- UI 没有 phase 就直接乱改
- 还没想清用户主任务就开始堆控件
- 对“暂时不做”的边界不够坚决

## 12. 我以后要保持的好习惯

- 每做完一段就记操作日志
- 每个改动都尽量形成小 commit
- 先跑回归再谈发布
- 先做实用，再做漂亮
- 项目结束后做复盘

## 13. 何时停止一个项目

- 主问题已解决
- 维护成本超过兴趣收益
- 新功能开始偏离主线
- 只剩零散需求，没有明确主价值
