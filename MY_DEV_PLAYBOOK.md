# MY_DEV_PLAYBOOK

## 1. 这份 Playbook 是干什么的

这不是“理想化的软件工程手册”，而是我在 `AION2 Dashboard` 这个项目里实际验证过、以后还会继续沿用的一套做项目方法。

它的目标只有一个：以后做新项目时，不再靠临时感觉推进，而是有一套可以重复使用的流程。

适用范围：

- 学习项目
- 自用工具
- 小型桌面应用
- 有明确用户场景的兴趣项目

## 2. 新项目启动前先做什么

开始写代码之前，先回答这 5 个问题：

1. 这个项目到底解决什么问题  
2. 谁会真正使用它  
3. 最小可用版本必须包含什么  
4. 明确不做什么  
5. 什么状态算“项目完成”  

如果这五个问题答不出来，就不要急着开写。

## 3. 新项目默认先建的 4 份文件

以后新项目一开始，先建这四份文件：

- `README.md`
- `FRAMEWORK.md`
- `OPERATION_LOG.md`
- `RELEASE_RUNBOOK.md`

原因：

- `README.md` 负责说明项目是什么
- `FRAMEWORK.md` 负责说明结构边界
- `OPERATION_LOG.md` 负责记录关键演进
- `RELEASE_RUNBOOK.md` 负责让发布流程可重复

没有这些文件，项目一旦做长，就会开始失忆。

## 4. 我以后默认采用的开发原则

### 4.1 先闭环，再扩功能

先把一个最小可用流程跑通，再考虑更多功能。  
没有闭环的功能列表，只会制造幻觉。

### 4.2 先收边界，再谈架构优雅

没有明确边界时，重构往往是无效努力。  
先知道什么做、什么不做，再决定结构怎么收。

### 4.3 先保可发布，再保功能丰富

一个项目只有在“能稳定发布”之后，才算真正开始可用。  
否则永远停留在开发机里。

这里的“可发布”不只包括“能打包”，还包括：

- 版本号是否清楚
- 发布前校验是否固定
- 安装版更新链路是否可靠
- 用户更新体验是否自然

### 4.4 不在主流程里塞低频工具

主任务页面应该优先服务最常见的动作。  
低频工具要后撤，不要抢主流程位置。

### 4.5 文档和日志跟着代码一起演进

文档不是最后补的总结，而是过程里的控制器。  
越晚补，越容易失真。

## 5. 项目结构基线

以后类似项目默认用下面这套分层：

`src/main`
- 持久化
- IPC 注册
- 系统资源访问
- 领域规则落地

`src/preload`
- 渲染层和主进程之间的安全桥

`src/renderer`
- 页面
- 组件
- 交互
- 轻量状态

`src/shared`
- 类型
- 常量
- 规则
- 契约

`scripts`
- 构建
- 校验
- 发布
- 回归辅助

这套分层的意义是：功能增长后，仍然知道“东西该放哪里”。

## 6. 需求推进方法

### 6.1 需求落地顺序

以后做功能，按这个顺序思考：

1. 用户的动作链路是什么  
2. 页面里的主任务是什么  
3. 哪些能力是低频的  
4. 最小可用实现是什么  
5. 先把最小闭环做完  

这个顺序比“我还能做什么”重要得多。

### 6.2 什么时候要冻结结构

这些时候必须冻结结构性重构：

- 架构重构刚做完
- 功能连续热修中
- 准备发版前

冻结后规则：

- 不再改整体分层
- 只做功能热修或局部增强
- 每个修复尽量一个小 commit

这条规则在 AION2 Dashboard 里非常有效，避免了边修边散。

## 7. 热修流程

以后所有热修默认按这个流程：

1. 明确问题  
2. 只改必要范围  
3. 改完立即跑：
   - `npm run typecheck`
   - `npm run test:unit`
4. 需要发版时再跑：
   - `npm run test:smoke`
   - `npm run dist:win`
   - `npm run check:release-yml`

热修说明必须写清楚三件事：

- 改了什么
- 为什么改
- 影响范围是什么

## 8. UI 工作方法

### 8.1 UI 不先谈“好看”

以后做 UI，第一步永远不是换颜色，而是判断：

- 信息架构对不对
- 主任务是不是在最前面
- 高级工具是不是后撤了

如果这三件事没做对，再好看的配色都只是装饰。

### 8.2 UI 改造顺序

以后大 UI 改造，按这套 phase 顺序做：

1. design tokens 收敛  
2. 信息架构重排  
3. 结果区优先级整理  
4. 导航和侧栏归位  
5. 全局一致性收尾  
6. 最后才做视觉抛光  

这套顺序在这个项目里已经验证过，比“直接大改页面”稳定得多。

### 8.3 判断 UI 是否变好的标准

不是看截图漂不漂亮，而是看：

- 打开页面时是不是先看到主任务
- 不读很多字能不能马上开始操作
- 高优先信息能不能一眼扫到
- 低频功能有没有抢视线

## 9. 发布流程

### 9.1 标准检查

以后发布前默认跑：

- `npm run typecheck`
- `npm run test:unit`
- `npm run build`
- `npm run test:smoke`
- `npm run dist:win`
- `npm run check:release-yml`

### 9.2 资产检查

发布时必须确认这些东西都在：

- Setup
- Portable
- 7z
- blockmap
- `latest.yml`

### 9.3 发布后验证

发布完不是结束，还要确认：

- 安装版能启动
- 版本号正确
- 自动更新链路正常
- `latest.yml` 的哈希和安装包一致

### 9.4 更新体验设计

以后只要项目准备长期使用，就尽量提前决定更新策略：

- 是手动检查更新，还是启动先检查
- 是后台下载后退出安装，还是启动即静默更新
- 更新失败时是阻断进入，还是允许继续进入旧版

我的默认建议是：

- 启动先检查
- 如果有更新，优先静默安装后再进入主界面
- 如果更新失败，允许继续进入旧版

原因很简单：

- 用户体验更接近成熟客户端
- 不需要用户理解“下载了但为什么还没装”
- 失败时又不会把用户锁死在启动阶段

## 10. 文档纪律

以后项目里要保持这些纪律：

- 大功能要有独立说明文档
- 每次发版都写 release notes
- 每个重要阶段都补 `OPERATION_LOG.md`
- 关键规则必须有基线文档

因为项目一旦持续几个月，真正会帮你记住上下文的，不是记忆，而是文档。

另外，更新体验如果发生关键变化，也应该写进：

- `README.md`
- `RELEASE_NOTES_vX.Y.Z.md`
- `OPERATION_LOG.md`

因为这类改动虽然不一定改变业务功能，但会显著改变用户感知。

## 11. 和 AI 协作的方法

### 11.1 适合交给 AI 的事情

- 代码实现
- 结构整理
- 回归执行
- 文档整理
- 发布执行

### 11.2 不该模糊表达的事情

- 版本边界
- 用户优先级
- 哪些功能绝对不做
- 发布策略

如果这些东西表达模糊，AI 很容易做出“看起来能做，但其实偏题”的结果。

### 11.3 高质量协作方式

以后和 AI 协作，尽量按这个格式给任务：

1. 先给目标  
2. 再给边界  
3. 再给验收标准  
4. 让 AI 小步推进、分 commit 落地  

这比“你看着改”高效得多。

## 12. 我要避免的坏习惯

以后要主动避免：

- 一边重构一边堆功能
- 没验证就发版
- UI 没 phase 就直接乱改
- 主任务还没想清楚就开始堆控件
- 对“暂时不做”的边界不够坚决

这些问题在这个项目里都出现过，所以以后必须主动规避。

## 13. 我要保持的好习惯

以后继续保持：

- 每做完一段就记日志
- 每个改动尽量形成小 commit
- 先跑回归再发版
- 先做实用，再做漂亮
- 项目结束后做复盘

这是当前最值得保留的习惯集合。

## 14. 何时停止一个项目

以后判断项目是否该停，不看“还有没有想法”，而看：

- 主问题是不是已经解决
- 维护成本是不是开始高于兴趣收益
- 新功能是不是开始偏离主线
- 需求是不是只剩零散修补

满足这些条件时，停止维护是合理的，不是半途而废。

## 15. 下一项目启动模板

以后新项目启动时，可以直接套这个模板：

- 项目目标是什么
- 核心用户是谁
- 最小闭环是什么
- 明确不做什么
- 分层怎么定
- 发布策略是什么
- 和 AI 的协作规则是什么

只要这几个问题先写清楚，下一个项目大概率会比这次更稳。

## 16. 一页启动清单

如果不想每次都翻完整手册，直接用这份一页清单：

- `NEXT_PROJECT_START_CHECKLIST.md`

它适合在项目真正开工前快速逐项确认，用来避免：

- 目标没定就开写
- 边界没定就扩功能
- 主任务没定就开始堆页面
- 更新策略没定，等发版时再临时补
