# NEXT_PROJECT_START_CHECKLIST

## 0. 这份清单怎么用

新项目开始前，按顺序把下面每一项填掉。  
填不出来的地方，不要急着开写代码。

## 1. 先判断值不值得做

- 这个项目解决的具体问题是什么？
- 这个问题是不是自己会长期遇到？
- 这个项目做出来后，自己会不会真的用？
- 它是学习项目、兴趣项目，还是准备长期维护的产品？

结论：
- 如果只是“看起来能做”，先不要做
- 如果问题真实、场景稳定、自己会用，再继续

## 2. 先写清楚边界

把下面几项写成一句话：

- 核心用户是谁
- 主任务是什么
- 最小闭环是什么
- 哪些功能明确不做
- 什么状态算项目完成

建议格式：

- 核心用户：
- 主任务：
- 最小闭环：
- 不做清单：
- 完成标准：

## 3. 先定页面和主流程

- 用户打开项目后，第一眼应该看到什么？
- 用户最常做的动作是什么？
- 哪些功能是低频工具，应该后撤？
- 是否存在“一个页面堆了太多功能”的风险？

检查标准：

- 主任务必须在首屏
- 低频功能必须折叠或后撤
- 不要一开始就做“大而全”的页面

## 4. 先定项目结构

默认结构：

- `src/main`
- `src/preload`
- `src/renderer`
- `src/shared`
- `scripts`

启动时先确定：

- 哪些逻辑属于主进程
- 哪些逻辑属于渲染层
- 哪些规则必须放进 shared
- 哪些发布/校验动作需要脚本化

## 5. 先创建基础文档

项目开始第一天就建：

- `README.md`
- `FRAMEWORK.md`
- `OPERATION_LOG.md`
- `RELEASE_RUNBOOK.md`

不要等后面再补。

## 6. 先准备发布思路

开始写代码前，就先回答：

- 这个项目将来怎么打包？
- 有没有版本号？
- 发布前最少跑哪些检查？
- 以后需不需要自动更新？
- 更新是“手动检查”还是“启动先检查”？
- 如果有更新，要不要静默安装后再进主界面？

即使第一版还不发布，也要先想清楚。

## 7. 先定义开发规则

默认规则建议直接采用：

- 先做最小闭环
- 架构重构后冻结结构
- 热修只做小 commit
- 每次改完跑 `typecheck + test:unit`
- 发布前跑 `test:smoke`
- UI 改造必须分 phase

## 8. 先定义 AI 协作规则

每次给 AI 任务时，至少说清楚：

- 目标是什么
- 边界是什么
- 哪些不要动
- 验收标准是什么
- 要不要直接改代码

如果你自己说不清这些，AI 也很难做对。

## 9. 先做最小版本，不做梦想版

第一版只做：

- 一个真实主任务
- 一条完整主流程
- 一套最基本的数据存储
- 一次可验证的输出结果

不要第一天就做：

- 全部高级功能
- 复杂设置页
- 过早的美化
- 大而全的抽象

## 10. 第一次可发布前的检查

至少确认下面这些事情：

- 主流程能闭环
- 数据能保存
- 关键规则已固定
- README 能说明项目用途
- 有最小发布流程
- 出现问题时知道怎么回滚
- 更新策略已经明确，不是发版当天临时决定

## 11. 什么时候暂停，先别写

如果出现下面任意情况，先停下来：

- 主任务说不清
- 边界说不清
- 页面开始什么都想放
- 还没闭环就想做第二系统
- 还没能发布就开始追求复杂重构

## 12. 开工前最后确认

开工前逐项打勾：

- [ ] 问题真实
- [ ] 用户明确
- [ ] 主任务明确
- [ ] 最小闭环明确
- [ ] 不做清单明确
- [ ] 分层方案明确
- [ ] 基础文档已建
- [ ] 发布思路明确
- [ ] 更新策略明确
- [ ] AI 协作规则明确

全部打勾，再开始写第一版代码。
